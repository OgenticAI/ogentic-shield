## Overview

This document describes how OgenticAI engages vertical-domain experts
(legal, clinical, finance) to annotate training examples for Sotto-tier model
fine-tuning — and, critically, the access boundary that prevents any customer
data from entering the training loop through this channel.

This is the architectural and contractual backstop for the claim:

> *"The expert feedback loop doesn't leak customer data into training."*

---

## The access boundary

**What experts can access:**

- Prompts and responses **generated by OgenticAI** on OgenticAI's internal eval
  harness
- Synthetic examples created by OgenticAI staff using public domain sources
- Public domain text (court opinions, SEC filings, JAMA open-access articles,
  etc.) curated by OgenticAI

**What experts cannot access:**

- Any Sotto Desktop installation (theirs or any customer's)
- Any customer telemetry or session data
- Any data originating from a user's chat, document upload, or clipboard
- Any production OgenticAI systems beyond the annotation interface

This boundary is enforced at two levels:

1. **Technical**: Expert annotators access only the annotation interface, which
   is served from OgenticAI's internal eval harness. They have no credentials
   for production Sotto systems, customer databases, or telemetry pipelines.
2. **Contractual**: The contractor agreement (see below) explicitly prohibits
   annotators from seeking, accessing, or retaining any data originating from
   a customer context, and requires immediate disclosure if they believe they
   have inadvertently encountered such data.

## Annotation protocol

### How examples are generated

Training examples originate from one of three sources:

| Source type | Origin | Allowed for training |
|---|---|---|
| Synthetic | Generated by OgenticAI staff or LLM, based on public domain patterns | Yes |
| Public domain | Curated from publicly available legal, clinical, or financial texts | Yes |
| Licensed | Obtained under institutional license (see `docs/PROVENANCE.md`) | Yes, per license terms |
| Customer | Originating from any Sotto user session or customer deployment | **No — prohibited** |

### How annotation tasks are delivered

1. OgenticAI staff create annotation batches in the internal eval harness.
2. Each batch contains examples of the approved source types above.
3. Annotators receive tasks via the annotation interface only.
4. Annotators label examples (e.g., "this text contains attorney-client
   privilege markers", "this text is safe for cloud routing") and optionally
   provide free-text rationale.
5. Completed annotations are reviewed by OgenticAI staff before inclusion in
   any training dataset.

### Provenance metadata

Every annotated example in OgenticAI's training datasets carries the following
provenance metadata fields:

| Field | Description | Example |
|---|---|---|
| `source_type` | One of: `synthetic`, `public_domain`, `licensed` | `public_domain` |
| `source_uri` | URL or internal ID of the source document | `courtlistener.com/opinion/1234567` |
| `annotator_id` | Pseudonymous annotator ID (e.g., `legal-expert-07`) | `legal-expert-07` |
| `annotation_date` | ISO 8601 date of annotation | `2026-05-14` |
| `batch_id` | Internal batch ID | `legal-batch-0023` |
| `reviewed_by` | OgenticAI staff reviewer ID | `oge-staff-03` |
| `customer_data` | Always `false` for any example in training | `false` |

These fields are stored alongside every example in OgenticAI's corpus registry.
In any future audit, an auditor can inspect the metadata for any training
example and verify it carries `customer_data: false` and a legitimate
`source_type`.

## Contractor agreement

Expert annotators are engaged as paid contractors under standard contractor
agreements that include:

- **Access restriction clause**: Annotators agree not to seek, access, copy,
  or retain any data originating from OgenticAI customer deployments, user
  sessions, or production telemetry.
- **Disclosure obligation**: Annotators agree to immediately notify
  security@ogenticai.com if they believe they have inadvertently encountered
  customer-origin data.
- **No-ownership clause**: All annotations are work-for-hire; OgenticAI retains
  full ownership.
- **Confidentiality**: Annotators agree not to disclose the contents of
  annotation tasks or OgenticAI's internal eval corpora.

A summary template of the access restriction and disclosure clauses is available
to enterprise customers on request (contact [legal@ogenticai.com](mailto:legal@ogenticai.com)).

## Audit trail

OgenticAI maintains an internal audit log of all annotation batches. The audit
log records:

- Which examples were in each batch
- Which annotators worked on each batch
- The source type and URI of every example
- The `customer_data` flag (always `false` for training examples)
- The date each batch was reviewed and approved by OgenticAI staff

This audit trail is retained for a minimum of five years. Enterprise customers
may request an attestation letter confirming that their deployment's data was
not used in any training batch, signed by OgenticAI's CTO.
