### Latency caveat

The ``run_benchmarks`` runner's median-latency target (<100ms) was
designed for the L1+L2-only path. With Layer 3 ON the median naturally
includes a model round-trip and is not directly comparable to the v0.1
target. Compare medians _across models_ here, not against 100ms.

## Discussion

### Headline finding

**No model — including the L1+L2-only baseline — meets the PRD §8 precision targets on the OGE-51 dataset as it stands today.** The targets (legal ≥90%, PHI ≥92%, MNPI ≥88%) reflect production aspirations, not the current v0.2 implementation. Treating them as v0.2 ship gates would block the milestone; treating them as v0.3 acceptance gates is realistic.

### What Layer 3 does to precision

Across every Layer-3-on row in every profile, **enabling the LLM degrades precision by 15–30 percentage points** vs. the L1+L2 baseline. Recall improves modestly (most visibly on `finance_mnpi`: 75% → 83%). The degradation pattern is the same in each profile: the LLM finds extra entity spans the L1+L2 stack correctly skipped. Looking at the per-recognizer tables in `run_layer3_benchmark.py` output (legal example): `WORK_PRODUCT` adds 4 FP, `COUNSEL_COMMUNICATION` adds 6 FP, `SETTLEMENT_TERMS` adds 1 FP. These are the categories where the LLM is most willing to "interpret" rather than match.

### Per-model takeaways

- **`granite3.1-moe:1b`** (current `fast` default) — 60 / 55 / 56% precision. Low latency (≤1.8s median). Eager classifier, lots of FPs.
- **`granite3-moe:3b`** — *worse* than 1B on finance (48% vs 56%). Bigger MoE doesn't help here; the model gets more confident about its hallucinations. Recommend **not** using 3B over 1B at the same quality tier.
- **`llama3.2:3b`** — best **active** model (67 / 65 / 67%). Still worse than L1+L2 baseline. Worth considering as a "balanced" tier if Sotto wants something marginally better than 1B without the qwen3 latency.
- **`qwen3:4b`** — superficially matches the baseline (86 / 85 / 75%), but the median latency of 60s+ on the therapy dataset and 19s+ on legal indicates the call timed out repeatedly (60 000ms timeout). When `OllamaClient.classify` hits its retry limit, it returns `None` and `run_layer3` falls back to the L1+L2 entity list — which is *exactly* what these qwen3 numbers reflect. The qwen3 row is a measurement of "Layer 3 timed out" not "Layer 3 with qwen3 worked well." Discounted.
- **L1+L2 baseline** — strongest *practical* configuration today. 86 / 85 / 75% precision. Recall is high on legal and PHI but middling on finance.

### Recommendations

1. **Keep `granite3.1-moe:1b` as the `fast` default in `ModelRegistry`.** It is functional, fast, and ships. Not because it improves precision — but because demoting it would imply we have a working Layer 3 default, which we don't.
2. **Drop `granite3-moe:3b` from the recommended set.** The 3B variant is not a Pareto improvement over 1B and isn't currently a candidate for any registry tier.
3. **Document that Layer 3 is opt-in and currently degrades precision.** The README should set expectations: enabling Layer 3 today buys you marginal recall at the cost of precision. Production callers should leave `enabled: false` until OGE-321 lands.
4. **The real fix is OGE-321 (calibration) plus prompt scope-tightening.** Two complementary follow-ups:
   - **Per-layer calibration** (OGE-321) lets a Layer-3 detection at confidence 0.9 carry less weight than a regex match at 0.9, so the score doesn't get pulled around by over-firing.
   - **Prompt narrowing**: restrict Layer 3 to *only* propose entities for categories the regex/NER layers haven't already covered. Most of the FPs in the table above are duplicates of categories L1+L2 already handles correctly.
5. **Raise the `qwen3:4b` timeout floor.** If anyone tries qwen3 again, set `timeout_ms ≥ 90 000` to actually capture its real precision/latency rather than the timeout-fallback signal.

### Methodology

```bash
ollama serve &
ollama pull granite3.1-moe:1b granite3-moe:3b llama3.2:3b qwen3:4b
.venv/bin/python benchmarks/run_moe_comparison.py \
    --json benchmarks/MOE_COMPARISON.results.json \
    --md   benchmarks/MOE_COMPARISON.md
```

Re-run from the repo root any time the prompts (`src/ogentic_shield/layers/llm_prompts.py`) or the calibration constant (`src/ogentic_shield/layers/llm_client.py:CONFIDENCE_CALIBRATION_FACTOR`) change. Commit the updated markdown so the `git log` shows precision movement over time.

Hardware: Apple Silicon, Ollama default `OLLAMA_NUM_PARALLEL=1`, `timeout_ms=60_000`, `ambiguous_score_range=[0, 100]` (forces Layer 3 on every example regardless of L1+L2 score).
